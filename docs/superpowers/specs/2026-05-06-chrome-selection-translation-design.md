# Chrome 划词翻译插件设计

## 概述

目标是从零创建一个 Chrome 插件，实现网页划词翻译。第一版聚焦最小可用闭环：用户在网页中划词后，可通过悬浮卡片或右键菜单触发翻译，插件自动识别中英文方向，并展示原文与译文。翻译能力接第三方翻译服务 API，密钥由用户在插件设置页自行填写。

## 已确认范围

- 平台：Chrome Extension，基于 Manifest V3
- 触发方式：
  - 默认自动弹出悬浮翻译卡片
  - 设置中可切换为“小按钮触发”
  - 同时支持右键菜单翻译入口
- 语言能力：
  - 自动识别语言
  - 至少支持英文到中文、中文到英文双向翻译
- 翻译结果：
  - 第一版只显示原文和译文
- API Key：
  - 由用户在插件设置页填写并保存

## 不在第一版范围内

- 音标、发音、例句、词典解释
- 多目标语言切换
- 登录账号、云同步、服务端代理
- PDF、图片 OCR、浏览器外应用取词
- 发布到 Chrome Web Store 的完整物料与文案

## 方案选择

### 候选方案

1. 直连单一翻译厂商 API
2. 插件内建立统一翻译适配层，再接具体 provider
3. 插件只做交互，请求全部走自建后端

### 选型结论

采用方案 2：建立统一翻译适配层。

原因：

- 第一版仍可直接在插件侧接第三方 API，开发成本可控
- 不把消息流和 UI 逻辑绑定到某一家厂商
- 后续切换 Google / DeepL / 有道等服务时，只需替换 provider 层

## 架构设计

插件拆分为四个核心部分：

### 1. Content Script

负责运行在网页上下文中，承担以下职责：

- 监听用户划词行为
- 读取当前选区文本与位置信息
- 判断当前触发模式
- 注入并控制小按钮与翻译卡片
- 与 background/service worker 通信
- 渲染加载中、成功、失败等状态

### 2. Background / Service Worker

负责插件全局逻辑：

- 注册右键菜单
- 接收来自 content script 或右键菜单的翻译请求
- 读取存储中的用户设置和 API Key
- 调用统一 provider 接口执行翻译
- 将结果发回当前标签页

### 3. Options Page

负责设置管理：

- 填写和更新 API Key
- 选择触发模式：
  - 自动弹出
  - 小按钮触发
- 显示基本状态提示，例如“已保存”

### 4. Storage

使用 Chrome 存储能力保存设置：

- `apiKey`
- `triggerMode`
- 可选的 provider 类型字段，为后续切换服务商预留

默认优先使用 `chrome.storage.sync`；若密钥同步策略受限或存在兼容性顾虑，可降级为 `chrome.storage.local`。实现上应通过统一存储模块屏蔽差异。

## 核心数据流

### 网页划词路径

1. 用户在页面中完成划词
2. content script 读取选中文本并做合法性校验
3. 若当前为自动模式，则直接发送翻译请求
4. 若当前为按钮模式，则先在选区附近显示一个小按钮
5. 用户点击小按钮后，再发送翻译请求
6. background 读取设置与 API Key，调用 provider
7. provider 返回译文结果
8. background 将结果发送回 content script
9. content script 在选区附近显示翻译卡片

### 右键菜单路径

1. 用户划词后使用右键菜单触发翻译
2. background 从右键菜单事件拿到选中文本
3. background 执行翻译请求
4. background 通知当前标签页展示统一样式的翻译卡片

## 组件与状态设计

页面内保留三个交互组件：

### 1. Selection Trigger

负责识别选区变化、文本合法性和触发模式决策。

### 2. Floating Action Button

仅在按钮模式下展示：

- 贴近选区显示
- 视觉上尽量轻量，避免遮挡正文
- 用户点击后触发翻译

### 3. Translation Popup

负责展示翻译反馈，包含三种状态：

- Loading：正在请求翻译
- Success：展示原文和译文
- Error：展示简短错误说明

卡片行为规则：

- 点击卡片外部区域关闭
- 用户重新划词时，旧卡片与旧请求状态失效
- 卡片只展示当前最近一次有效选区的结果

## 触发与过滤规则

为减少误触和噪音，第一版加入以下规则：

- 空字符串或纯空白不触发
- 超长文本不触发，长度阈值在实现中集中配置
- 不规则多段选区不处理，优先支持单连续文本
- 用户快速重复触发同一文本时，可复用最近一次翻译结果
- 新请求发起后，旧请求结果若晚到，应被丢弃，避免覆盖最新状态

## 翻译 Provider 抽象

定义统一翻译接口，供 background 调用：

- 输入：
  - `text`
  - 可选 `sourceLang`
  - 可选 `targetLang`
- 输出：
  - `sourceLang`
  - `targetLang`
  - `originalText`
  - `translatedText`

要求：

- provider 内部处理具体鉴权方式和 HTTP 请求结构
- UI 与业务主流程不依赖某个厂商特有字段
- 中英文双向的自动识别结果以 provider 返回为准，前端不做二次猜测

## 设置页设计

第一版设置页包含最少字段：

- API Key 输入框
- 触发模式切换
- 保存按钮
- 保存结果提示

交互要求：

- 未填写 API Key 时允许保存空值，但翻译时必须给出明确引导
- 设置修改后立即写入存储
- content script 与 background 读取统一配置源，不维护各自副本

## 权限与 Manifest 方向

Manifest V3 下至少需要考虑：

- `storage`
- `contextMenus`
- `activeTab`
- 与目标翻译 API 域名对应的网络访问权限

页面注入方式以 content script 为主，避免不必要的高权限申请。权限声明应只覆盖实际使用范围。

## 错误处理

第一版明确覆盖以下错误：

### 1. 未配置 API Key

- 不发送请求
- 提示用户前往设置页填写 API Key

### 2. API Key 无效或鉴权失败

- 向用户显示简短错误
- 不暴露厂商原始报错细节

### 3. 网络失败或超时

- 显示“网络异常，请稍后重试”类提示

### 4. 页面受限

- 对于受限页面，悬浮 UI 可能不可用
- 右键菜单能力尽量保留
- 需要在实现时针对 Chrome 特殊页面做降级处理

### 5. 语言识别异常

- 以 provider 返回为准
- 若返回缺失，则以通用失败处理展示，不在前端兜底猜测语言

## 测试策略

### 手工验证

覆盖以下场景：

- 普通网页中的英文到中文翻译
- 普通网页中的中文到英文翻译
- 自动模式
- 小按钮模式
- 右键菜单触发
- 未填写 API Key
- 错误 API Key
- 网络失败或超时
- 重复快速划词

### 单元测试

重点覆盖：

- 设置读写模块
- provider 适配层
- 文本过滤规则
- 触发模式决策逻辑

### 集成验证

验证主消息链路：

- `content script -> background -> provider -> content script`

确认状态切换、消息结构和错误返回格式一致。

## 实现边界与拆分建议

代码层面建议拆成以下模块：

- `manifest` 配置
- `content/selection`：划词监听与坐标计算
- `content/ui`：按钮与弹层渲染
- `background/translate`：消息处理与请求编排
- `providers`：第三方翻译服务适配
- `options`：设置页逻辑
- `shared/storage`：统一配置访问
- `shared/types`：消息体与翻译结果类型

这种拆分有两个目的：

- 保持内容脚本、后台脚本和 UI 逻辑边界清晰
- 让后续更换 provider 或扩展结果字段时不需要重写主流程

## 成功标准

完成后，第一版应满足以下结果：

- 用户安装插件后可在设置页填写 API Key
- 用户在普通网页中划词时，可按设置触发翻译
- 自动识别中英文方向并返回译文
- 悬浮卡片只展示原文和译文
- 右键菜单可作为备用入口正常工作
- 常见错误场景有明确反馈，不出现无响应状态
